# 程序员四象限与 LLM 时代的职业分化

Last Updated: 2026-04-15
Topic Summary: 一篇未来 blog 的素材——把"程序员工作 = 业务理解 + 程序设计"这个能力二分推进到"领域专家 / 技术专家"两条职业路线，再推进到四象限模型。核心命题：LLM 精确冲击四象限中的"左下角"（业务浅 + 技术浅的中间地带），而这个地带恰好是过去 30 年程序员就业市场的最大池子。未来只剩三个角（系统专家 / 领域专家 / 架构师），每个角都要求极深的单侧能力，"均衡型程序员"没有位置。

状态：**素材记录，未动笔**。当前正在写的 blog 是从系统架构师视角讲"Agent 工程重演计算机系统史"，与本文互补但不合并。

---

## 一、起点：能力二分

程序员的工作由两个独立部分组成：

1. **业务理解**——把世界的一块（电商/金融/医疗）翻译成可计算结构
   - 领域建模
   - 约束采集（政策、监管、历史）
   - 隐式知识显式化
   - 抽象层级选择

2. **程序设计**——驯服机器让它正确、高效、可演化地执行
   - 数据结构/算法（皮）
   - 抽象设计（命名、分层、依赖）
   - 状态管理
   - 错误处理
   - 并发/分布式
   - 可观测性
   - 可演化性

**陷阱提醒**：
- 不要把"业务理解"写成"和 PM 扯皮"——它是真正的工程活（Eric Evans DDD）
- 不要把"程序设计"写成"算法+设计模式"——范围大得多

---

## 二、从能力到路线：两个层次

```
能力层：    业务理解        程序设计
             ↓               ↓
职业层：    领域专家        技术专家
```

能力是每个程序员身上都有两半（只是比例），路线是生涯漂移方向。两层不要混。

**"两条路线"这个命名来自大公司的 IC vs Manager / P 序列 vs M 序列**。但如果 blog 只停在这一层，会落到 HR 咨询文的水位。

---

## 三、真正的抽象：四象限

|  | **技术深度高** | **技术深度中低** |
|---|---|---|
| **领域深度高** | **架构师型**（两半都强，稀缺） | **业务专家 / PM 方向** |
| **领域深度中低** | **系统专家**（造底层/工具/基础设施） | **普通程序员**（最大池子，被 LLM 精确冲击） |

四象限比两条路线解释力更强，因为它回答了一个关键问题——**LLM 精确冲击的不是"程序员"大类，而是左下角**。

### 各象限的 LLM 耐受性

- **右上（架构师）**：LLM 替代不了，反而放大——做"把具体业务翻译成正确系统结构"这件事
- **右下（业务专家/PM）**：LLM 没直接冲击，他们本来就不写代码
- **左上（系统专家）**：LLM 替代不了，造 LLM/infra 本身需要他们
- **左下（普通程序员）**：LLM 直接替代目标——两半都浅

---

## 四、核心命题：左下角是 30 年就业市场的主体

大量"做业务系统的程序员"——CRUD、内部工具、企业集成——**业务懂一半、技术懂一半，两半都不深**。

公司养得起，因为他们：
- 比纯业务人员便宜
- 比架构师便宜
- 数量足够大能批量招

**左下角是"中产程序员"的位置**，也是过去 30 年 CS 教育的批量产物。

**LLM 能做的事精确对应左下角产出**：
- 写浅代码（CRUD / boilerplate / 标准业务逻辑）✓
- 理解具体公司业务内核 ✗（需要右下）
- 设计底层系统 ✗（需要左上）
- 把模糊需求翻译成系统设计 ✗（需要右上）

**所以 LLM 不是淘汰程序员，是淘汰中间地带**。

## 五、撤离策略：左下角往哪走

"两条路线"的真正意义不是职业选择题，是**生存策略**——往哪个角撤：

```
左下角 (被冲击) ─┬─→ 右下角: 往业务深处扎（领域专家）
                  └─→ 左上角: 往技术深处扎（系统专家）
```

两条路线**看似对称，其实不对称**：

| | 右下（领域专家） | 左上（系统专家） |
|---|---|---|
| 进入门槛 | **低**（接触业务就攒） | **高**（数学/系统底子） |
| 时间回报 | 10-20 年才值钱 | 3-5 年看出差距 |
| 可迁移性 | **低**（换行业重学） | **高**（跨行业通用） |
| LLM 时代稳定性 | **极稳**（LLM 进不去公司内部） | **稳**（但要持续卷） |
| 天花板 | CTO / 业务 VP | Principal Engineer / 技术合伙人 |
| 失败模式 | 业务消失（行业衰退） | 技术过时（新范式来） |

**不对称的含义**：
- 右下门槛低但窗口期短——35 岁前没在行业扎 10 年，之后很难建起来
- 左上门槛高但贬值慢——底层系统知识 20 年半衰期
- 大多数左下"被迫选"右下，因为左上进不去
- 少数真选左上的人（有 CS 底子、愿意学底层）是 LLM 时代最稀缺的

---

## 六、右上角（架构师）：最难但最值钱

四象限里**右上才是天选之子**——两半都深。

稀缺不是因为难学，是因为**两边学习曲线不兼容**：
- 学好业务要在一家公司扎 10 年
- 学好技术要经常换栈/问题域

**"两半都强的人"违反自然成长路径**。他们的出现要么靠早期有意识规划，要么靠极少数公司提供足够好的环境（业务稳定 + 技术先进）——Google 早期、Netflix、Stripe 这一类。

**右上角在 LLM 时代会更值钱**：
- LLM 把左下变便宜 → 雇 1 个右上 + 一堆 LLM 比雇一群左下划算
- 右上是唯一能**判断 LLM 产出对不对**的人——同时懂业务上什么对、技术上什么优

## 七、Blog 可能的收敛结构

```
第一层：业务理解 + 程序设计（能力的两半）
第二层：领域专家 + 技术专家（路线的两条）
第三层：四象限（真正的抽象）
第四层：LLM 让中间地带消失 ← punchline
第五层：三个角的生存策略不对称
```

**真正的主题（而非表面的"两条路线"）**：

> 70 年来"程序员"这个职业能存在，是因为中间地带（左下角）养活了大多数人。LLM 精确吃掉这个地带。未来只剩三个角，每个角都需要极深的单侧能力——两半浅浅都会的"均衡型程序员"没有位置了。

## 八、几个值得放进去的反直觉命题

让 blog 有记忆点的候选金句：

1. **"程序员的 70 年是不断把程序设计这一半外包给机器的过程"**——汇编器外包了指令、编译器外包了寄存器分配、GC 外包了内存管理、框架外包了架构、LLM 外包了编码。每次"被淘汰"都意味着业务理解那一半价值更凸显。

2. **"软件危机从来都是业务危机的伪装"**——所谓"软件难写"往往不是技术难，是业务没想清楚。几十年的 SE 方法论大多在用工程手段补救业务侧的混沌。

3. **"CS 学的东西 90% 属于程序设计那一半，所以 CS 毕业生到公司第一年的痛苦 90% 来自业务理解那一半"**——学校不教第二半。为什么不教？因为第二半不能抽象成课本。

4. **"DSL 是业务人员写程序的永恒梦想，永远失败"**——因为业务人员不想学任何形式语言。LLM 是第一个真正可能让这个梦想成立的东西，但成立方式是"LLM 替他们写"而不是"他们学会写"。

5. **"程序员最值钱的能力是'听懂非程序员说话的能力'"**——精确描述业务翻译这半的本质。LLM 还没追上。

6. **"LLM 不是淘汰程序员，是淘汰均衡型程序员"**——这才是真命题。

## 九、写作注意

- **长度**：6000-8000 中文字（按 blog.1 体量估）
- **起头**：用具体故事比定义好——比如一个程序员 1985 vs 2025 同一天的工作对比
- **结尾**：**给观察不给建议**。建议显得功利，观察显得通透。让读者自己对号入座。
- **避免**：
  - 不要列"程序员必学的 N 项技能"
  - 不要做预测清单
  - 这些会让 blog 变公众号文章
- **政治敏感性**：**"均衡型程序员没有位置"会引起反感，但它真实**。用描述语气（"这件事正在发生"），不用建议语气（"你应该选"）。
